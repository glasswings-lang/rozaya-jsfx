# v2.9 regression: render and alt-tab resume use stale plugin state

Found in Melody Phase during the v2.10 work. Likely affects every plugin in the suite — all of v2.9+ has the same shape that introduces the bug.

This note is self-contained so a fresh session can pick it up cold.

---

## ✅ STATUS 2026-06-15: FULLY RESOLVED + PROPAGATED SUITE-WIDE. Fix = **Path A** (render) **+ serialize version guard** (where a serialized count changed) **+ Park-Miller PRNG** (a separate bug found mid-sweep). Every active plugin done and ear-tested by Rozaya (live clean, render-from-0, alt-tab-from-0; config survives stop/play; where applicable, config survives save/reload). The two legacy wombs (v1/v2) were archived frozen-as-shipped instead of fixed.

Per-plugin commits on branch `fix/v2.9-render-and-serialize` (not yet merged/tagged at time of writing):
- **Render (Path A) — all active plugins:** melody, polyrhythm, breath, heartbeat, womb v3, rhythm-track, shepard-scale, shepard-tone, tremolo, sweeping-filter, sweep-dwell, resonance-bank.
- **Serialize version guard — only where a serialized count changed across (or within) a release:** melody (10→28 targets, dev-only exposure) and polyrhythm (13→24, pre-v2.10). resonance-bank got a *new* versioned `@serialize` (it had none — closes a pre-existing persistence gap). All other plugins' serialize formats never changed, so no guard added (adding one would needlessly reset existing drift configs).
- **Park-Miller PRNG (separate bug) — plugins with a hand-rolled R-channel LCG:** breath, heartbeat, womb v3. The decorrelated noise channel used multiplier 22695477, which overflowed EEL2's float64 2^53 integer limit and degenerated into a periodic tone + DC offset on R only (audible "swung" wobble that scaled with volume and survived into renders). Replaced with Park-Miller / MINSTD. See the CLAUDE.md gotcha "LCG random/noise generators overflow EEL2's 2^53 integer limit."

The historical analysis below is preserved for context. Everything from "## User-visible scenarios" down is the original cold-open writeup.

**The key discovery: there were TWO independent bugs, and the second was masking the first.**

1. **The render bug was real, and Path A (commit `5d1134e`) is the correct fix for it.** Drop `ext_noinit`, let REAPER re-run `@init` on every transport play / render start (stock-filter behavior), guard ONLY the drift CONFIG bank behind a one-time `drift_cfg_inited` flag so the re-run doesn't wipe it, and delete the explicit `@sample` edge handler. REAPER *does* re-run `@init` on an offline render's transport start (the open question below is now answered: **yes**). This restores the v2.8 render behavior and also fixes alt-tab resume.

2. **Path A's apparent "it didn't work" was a SEPARATE serialize-migration bug contaminating the render test.** Melody's drift target count grew 10 → 28 (commit `57671a3`) without versioning the `@serialize` format. The stream began directly with `target_drift_up[0]`, so loading a 10-target project (41 floats) with 28-target reads ran the sequential `file_mem` cursor *past each array into the next*, scrambling drift across targets and **fabricating phantom drift** on Rate Value (target 0) + V1 Timing (target 1) + the note-duration targets — even though the user had configured **zero** drift. Symptom: notes arriving off-time / tempo lurching on live output, and every render sounding wrong regardless of stale state. This is why Path A "didn't fix anything" — the render *was* starting from 0, but the invented Rate-Value drift made it sound broken anyway. It only surfaced "now" because installing the Path A build was the user's first jump to a 28-target plugin against a 10-target-saved project.

   **Fix:** lead the `@serialize` stream with a count-encoded magic (`1000000 + N_TARGETS` = `1000028`). On load, if it doesn't match this build's value the blob is a different/older format and we leave drift at the `@init` defaults (drift off). A legacy blob's first float is `up[0]` ∈ [0,20], which can never equal the magic, so old projects deterministically fall through to defaults — consistent with the suite's accepted "old drift configs reset, reconfigure" migration policy. See the `@serialize` block in `src/melody_phase.jsfx`.

**Path B (commit `c3c79f6`) remains known-bad and was correctly rejected.** Broadening the transport-edge reset to fire on the STOP edge zeroed voice envelopes mid-decay → ringing release tails cut to silence → "crunchy / broken last 4 notes." Do NOT reset runtime state on the stop edge for any plugin with release tails. Path A needs no stop-edge handler at all.

### Propagation checklist (both fixes, suite-wide)

Two distinct changes to carry to the other plugins. They are independent — apply both:

- **Render fix (Path A shape):** drop `ext_noinit`; guard the per-target drift CONFIG init behind a one-time `*_cfg_inited` flag (so `@init` re-running on play doesn't wipe it) while leaving drift RUNTIME + sequencer/voice/gate state in the unguarded `@init` path; delete the explicit `@sample` transport-edge reset handler and its `last_play_state`. Applies to every v2.9+ plugin that has `ext_noinit = 1`.
- **Serialize version guard:** any plugin whose serialized target/field count grew *after* it first shipped a serialize format is vulnerable to the same misread. Audit each plugin's `@serialize` against the format its already-saved projects used. Add the `magic = 1000000 + <count>; file_var; if-match` guard. Plugins that have only ever had one serialize format still benefit (cheap insurance against future count changes) but are not currently broken.

Verify each plugin by ear on Melody Phase's three checks (clean live output, render-from-0, alt-tab-from-0) before moving to the next. Two earlier failed attempts say: do not propagate on reasoning alone.

The original analysis + the two-path writeup below is preserved as reference. Treat its "Recommended approach: Path B" as **superseded — Path B is known-bad; Path A + serialize guard is the answer.**

## User-visible scenarios (both broken)

1. **Render after stop.** Play → Stop → Render. The plugin's sequencer / oscillator phases / drift runtime / state machine all keep wherever they were when the user pressed stop. Render captures mid-sequence audio, not "starting from 0."
2. **Alt-tab away during stopped state, then back.** Play → Stop → alt-tab away to another app → wait → alt-tab back to REAPER. Sound resumes from the exact sample the audio engine paused at.

## Works correctly

Plain "play after stop" — sequence does start from 0. The user explicitly confirmed this works.

## This is a v2.9 regression

At **v2.8** (the Speed Ramp sweep tag, immediately before the drift work), plugins did NOT have `ext_noinit = 1`. REAPER's default behavior re-ran `@init` on every transport play, which automatically reset everything. The transport-edge handler added during the Speed Ramp work was technically redundant with `@init`'s auto-reset — but the redundancy worked in our favor. Every transport state change triggered a reset via one path or the other. **Render-after-stop worked at v2.8 because `@init` fired on render start.**

The very first commit of the v2.9 drift-sweep on each plugin added `ext_noinit = 1` specifically to preserve the per-target drift CONFIG bank across transport play (so users don't have to reconfigure drift every play press). For breath_gen this was commit `e9db305 feat(breath-gen): drift → nested-selector pattern`; other plugins got it in their respective drift-sweep commits.

The reasonable assumption at the time: the existing explicit transport-edge handler covers what `@init`'s auto-reset used to do, so disabling `@init`-on-transport via `ext_noinit` is safe. **It isn't.** The edge handler only fires on `play_state 0 → 1`, which doesn't catch:

- Render starts (depending on how REAPER's offline render takes the transport state through transitions visible to JSFX).
- Alt-tab resume (`play_state` never changes during alt-tab pause — audio engine pauses without changing transport state).

`@init`'s auto-reset was catching all those silently because it ran whenever transport state changed for ANY reason.

## Compare to REAPER's stock sweeping filter

At `<reaper config dir>/Effects/filters/sweepinglowpass`:

```jsfx
@init
  spos=0; israte2=2/srate;

@slider
  res_lp=slider4;
  fcenter=(slider1+slider2)*0.5;
  fsize=(slider1-slider2)*0.5;
  ssc=$pi*israte2 / slider3;

@sample
cut_lp = (fcenter + cos(spos*ssc)*fsize) * israte2;
spos=spos+1;
fb_lp=res_lp+res_lp/(1-cut_lp);
n3=n3+cut_lp*(spl0-n3+fb_lp*(n3-n4));
n4=n4+cut_lp*(n3-n4);
spl0=n4;
rn3=rn3+cut_lp*(spl1-rn3+fb_lp*(rn3-rn4));
rn4=rn4+cut_lp*(rn3-rn4);
spl1=rn4;
```

No `ext_noinit`. No transport-edge handler. Just `@init { spos=0; }` with `spos` incremented per sample. Every transport play re-runs `@init`, resets `spos`, sweep starts fresh. **Render works correctly.** Simple because REAPER's default is right for stateless filters — but our drift bank can't live with that default without restructuring (see Path A below).

(Note: REAPER's stock filter probably has the same alt-tab behavior we do — audio engine pauses, `spos` freezes in memory, audio resumes mid-cosine. It just isn't audible there because a frequency-sweep filter has no quantized structure. Melody Phase's sequencer makes the same underlying behavior loudly audible.)

## Two fix paths

### Path A — matches REAPER stock filter philosophy

Drop `ext_noinit = 1` from every plugin. Let `@init` re-run on every transport play.

**Cost:** drift CONFIG bank needs different storage so `@init` doesn't wipe it. Options:

- Move bank handling into `@serialize` only (works for project save/load but not for the first-time-configure case unless we add an explicit "initialized" flag in user-facing slider state somewhere).
- Use a high memory region that `@init` deliberately doesn't `memset` (rely on REAPER not clearing arbitrary mem between calls).

Doable but a bigger refactor — every plugin needs the bank handling restructured. ~10 plugins × non-trivial-per-plugin work.

### Path B — minimal: keep `ext_noinit`, add a STOP edge handler

Add a SECOND reset edge on STOP (`play_state == 0 && last_play_state > 0`). Reset runtime state on both 1 → 0 AND 0 → 1 transitions. After stop, plugin is already in fresh state in memory. Render-after-stop captures fresh state. Alt-tab-then-back resumes audio from fresh state.

Code pattern (in `@sample` of each plugin, near where the existing transport-edge handler lives):

```jsfx
// STOP edge — reset runtime state when transport stops, so that
// render-after-stop and audio-engine-pause-resume both start fresh.
// v2.9 ext_noinit means @init no longer auto-resets on transport play;
// we have to catch the cases it used to via explicit edge handlers.
play_state == 0 && last_play_state > 0 ? (
  // (same reset block as the existing PLAY edge handler)
);

// PLAY edge — existing reset (still wanted for the first-play case
// after plugin instance creation, where last_play_state was init'd
// to 0 in @init but no stop has happened yet).
play_state > 0 && last_play_state == 0 ? (
  // reset runtime state
);

last_play_state = play_state;
```

Simplest implementation: factor the reset body into something both edges can call, OR duplicate the body. Each plugin's reset block is ~20-60 lines — duplication is uglier but readable.

## What neither path fixes

"Render while still playing" — if the user hits Render without stopping first, `play_state` stays at 1 throughout, no transitions occur, neither handler fires. **User discipline for reproducible renders stays: stop first, then render.** This should be documented in the manual under each plugin.

## Recommended approach

**Path B as the fast fix to unblock users.** Path A as a possible future refactor if other `ext_noinit`-related issues surface.

## Why it's likely suite-wide

All v2.9+ plugins have `ext_noinit = 1` and the same single-edge handler shape (the drift-sweep work added `ext_noinit` plus a single-edge handler in lockstep across all plugins). If Melody Phase has the bug, the others very likely do too. Melody Phase was first audibly noticed because its sequencer state is the most obvious mid-thing — heartbeat picking up mid-beat, breath mid-inhale, polyrhythm mid-voice-cycle, etc. would all be silently broken the same way.

## History trail

- **v2.8** (`dc0a41b` tag): Speed Ramp signed-delta sweep + transport-edge handlers added. No `ext_noinit`. Render worked because `@init` re-ran on transport play.
- **v2.9** (multiple commits, ending at `722b991` "drift nested-selector sweep SHIPPED"): added `ext_noinit = 1` to every plugin as part of preserving drift CONFIG bank across transport play. Render regression introduced here.
- **v2.10** (uncommitted at time of this note): Polyrhythm Phase expressive drift expansion to 24 targets. User noticed the render bug while testing Melody Phase during this work.

The fix can land as v2.10 alongside the polyrhythm expressive drift work, or as a separate v2.10/v2.11 release — author's call.
